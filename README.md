# IsdaSure

<p align="center"><strong>On-chain micro-insurance for coastal communities тАФ Soroban smart contracts on Stellar Testnet.</strong></p>

---

## Overview ЁЯзн

IsdaSure is a community-driven micro-insurance dApp that enables fisherfolk to pool small daily contributions into a shared on-chain fund. When a storm or extreme weather prevents fishing, the pooled funds are automatically distributed to contributors using a Soroban smart contract. The system prioritizes transparency, low friction, and accessibility for non-technical users.

Key goals:
- ЁЯТ╕ Low-cost, frequent micro-contributions
- ЁЯФН Transparent, auditable on-chain accounting
- ЁЯдЭ Community-controlled storm-trigger and fair payout distribution
- ЁЯФР Wallet-based UX via Freighter (signing) and lightweight backend for state aggregation

---

## Problem тЭЧ

Small-scale fisherfolk depend on daily income from fishing. When storms occur they lose earnings and often resort to high-interest loans or delayed aid. Traditional insurance products are unaffordable or inaccessible. Communities need a simple, trustless system to pool tiny contributions and distribute funds quickly during no-fishing days.

---

## Solution ЁЯТб

IsdaSure uses Soroban smart contracts to store contributor state and execute payout logic when a community-authorized trigger occurs. The frontend integrates with Freighter to sign transactions; the backend provides a file-backed state and graceful fallbacks so the UX remains friendly even when RPC or network issues occur.

Benefits:
- Immediate, transparent distribution of pooled funds when a storm is triggered.
- Minimal friction: small daily contributions, Freighter signing, and straightforward join/contribute flows.
- No intermediaries тАФ on-chain settlement and verifiable history.

Step-by-step explanation (long form):

1) Register & group setup ЁЯз╛
   - ЁЯУЭ User registers via the frontend (email/name or quick guest mode). Backend stores the record
     in `backend/data/users.json` and returns a `userId`.
   - ЁЯПШя╕П A user can create or join a group. Groups are saved to `backend/data/groups.json`.

2) Prepare contribution (frontend -> backend) ЁЯФз
   - ЁЯУи When a user wants to contribute, the frontend calls POST `/contribute/prepare` with
     `groupId`, `amount`, and the currently connected wallet address.
   - тЪЩя╕П Backend (`sorobanService`) tries to build a Soroban prepared transaction using the
     configured `SOROBAN_RPC_URL` and the contract's `contribute` method.
   - тЬЕ If the RPC and contract are healthy, backend returns an unsigned prepared XDR.
   - тЪая╕П If prepare fails (account not found, method missing, DNS error), backend returns a
     `mockPreparedTx` or a minimal `manageData` fallback payload and sets `mode=fallback`.

3) Signing (wallet) ЁЯФП
   - ЁЯФР The frontend asks Freighter to sign the prepared XDR (or fallback `manageData`). This ensures
     the wallet holder approves the action and the UX shows the expected signing flow.

4) Submit (frontend -> backend) ЁЯУд
   - ЁЯУм Signed XDR is POSTed to `/contribute`. Backend attempts `submitSignedSorobanTransaction`.
   - тЬЕ On success: backend polls for final status, updates `pool.json`, and adds an onchain entry
     in `chainHistory` with an explorer URL.
   - тЭМ On failure (or when in fallback/mock mode): backend records a mocked confirmation
     entry containing a `note` explaining why it was mocked. The UI shows success but marks
     the entry as `mock` for transparency.

5) Storm trigger & payouts тЫИя╕П
   - ЁЯСд An authorized admin triggers a storm via POST `/triggerStorm`.
   - ЁЯзо Backend calculates payout per eligible contributor (split by contribution shares).
   - тЬЕ If onchain: backend prepares `distribute_payouts`, admin signs+submits, and the contract
     performs transfers; backend stores the confirmed tx.
   - тЪая╕П If RPC unavailable: backend writes mocked payouts to state and records `chainHistory` notes.

6) UX & transparency ЁЯСА
   - ЁЯУК Contributions and payouts are visible in the UI. Mock confirmations are clearly marked
     with a `mock` note and may include an explanatory `note` or placeholder explorer link.
   - ЁЯз░ Admin dashboard shows seeded groups (copied from packaged JSON to `/tmp` on hosted runtimes).

7) Production notes & moving to real on-chain mode ЁЯЪА
   - тЪЩя╕П To enable real on-chain behavior set `SOROBAN_RPC_URL` and `SOROBAN_CONTRACT_ID` in Vercel.
   - ЁЯТ░ Ensure admin wallet is funded on testnet so prepares/submits succeed and `account not found`
     errors are avoided during prepare.

This flow is intentionally tolerant: contributors always sign something (either prepared XDR or a
manageData fallback) and the backend records a clear audit trail whether the result was onchain or mocked.

## Smart Contract: Build / Test / Deploy (contract/) ЁЯжА

1. Install the wasm target and build toolchain

```bash
rustup target add wasm32-unknown-unknown
```

2. Build the contract

```bash
cd contract
stellar contract build
```

3. Run tests

```bash
cargo test
```

4. (Optional) Deploy to testnet with Soroban / Stellar CLI

```bash
stellar keys generate --global my-admin-key --network testnet
stellar keys fund my-admin-key --network testnet
cd contract
stellar contract deploy --wasm target/wasm32-unknown-unknown/release/isdasure.wasm \
  --source my-admin-key --network testnet
```
